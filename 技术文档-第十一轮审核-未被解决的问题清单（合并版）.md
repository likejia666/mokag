### 问题17：消息删除与消息撤回概念混用【遗漏问题】
- **严重程度**：P0
- **所在章节**：
  - WebSocket协议：第158、163、165、277-286行（消息撤回协议）
  - WebSocket协议：第288-297、409-420行（消息删除协议）
  - 数据库：第985-1010行（chat_messages表）
  - 页面设计：第2739、2812行（长按操作菜单）
  - API说明：第9088-9091行（删除接口说明）
- **原文要点**：
  1. WebSocket有完整的两个协议：
     - `message.recall`（撤回）：推送 `message.recalled`，超时2分钟（9008错误）
     - `message.delete`（删除）：推送 `message.deleted`
  2. 数据库字段只有：
     - `is_deleted_sender`（发送者删除）
     - `is_deleted_receiver`（接收者删除）
     - `is_deleted_both`（双方删除）
     - **没有 `is_recalled` 撤回字段**
  3. 页面设计（2739行）：长按操作菜单只有"删除消息"选项，**没有"撤回消息"选项**
  4. API说明（9090行）：**发送方删除后，接收方查看时显示"[消息已撤回]"**
- **问题描述**：
  1. **概念定义混乱**：
     - "撤回"和"删除"是两个不同的操作
     - 撤回：发送方在一定时间内（2分钟）撤回报文，接收方看到"[消息已撤回]"
     - 删除：双方各自删除本地视图，不影响对方
  2. **数据库字段缺失**：
     - `chat_messages` 表没有 `is_recalled` 字段
     - 无法记录消息是否被撤回
  3. **页面设计缺失**：
     - 长按菜单只有"删除"，没有"撤回"入口
     - 用户无法执行撤回操作
  4. **术语混用**：
     - "删除消息"后，接收方却显示"[消息已撤回]"
     - 删除≠撤回，视觉反馈与操作不匹配
- **冲突说明**：
  - WebSocket协议定义了两个独立动作，但页面只实现了"删除"
  - 数据库缺少 `is_recalled` 字段，无法区分撤回状态
  - 删除操作显示为"[消息已撤回]"，逻辑矛盾
- **超计划设计**：
  - 设计了撤回协议但未在页面提供入口
  - 设计了删除显示规则但与操作名不匹配
- **影响**：
  - 用户无法撤回已发送的消息
  - 术语不一致导致用户困惑（删除≠撤回）
  - 实现时无法正确区分两种状态
- **建议修正**：
  1. 数据库：chat_messages表增加 `is_recalled` 字段（TINYINT，0否/1是）和 `recalled_at` 字段
  2. 页面：在长按操作菜单增加"撤回消息"选项（仅发送方可见，且消息发送时间在2分钟内）
  3. 显示规则明确区分：
     - 撤回消息：双方都显示"[消息已撤回]"，内容不可见
     - 删除消息：仅本端不显示，不影响对方
  4. 删除API说明中移除"[消息已撤回]"的显示规则（那是撤回的显示）
- **关联修改**：

  - **位置1：数据库chat_messages表（第985-1010行）**
    - 当前：
      ```
      is_deleted_sender     TINYINT  发送者是否删除：0否/1是
      is_deleted_receiver   TINYINT  接收者是否删除：0否/1是
      is_deleted_both       TINYINT  双方均已删除
      ```
    - 修改为：

      is_recalled          TINYINT  是否已撤回：0否/1是【新增】

  - **位置2：页面设计-长按菜单（第2739行）**

    - 修改为：
