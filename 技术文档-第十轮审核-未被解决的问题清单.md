## 问题一：职位报名状态机设计存在逻辑漏洞

**所在章节**: 3.9 报名表(applications) / 5.5 职位报名与审核流程图

**原文要点**:
```
状态流转：pending(待审核) → approved(已通过)/rejected(已拒绝)
取消报名：approved状态可取消 → 状态改为cancelled
恢复规则（5.5流程图）：取消后若仍有未满员工种/级别，职位可恢复为active
```

**问题描述**:
1. 文档在5.5流程图中描述"取消后若仍有未满员工种/级别，职位可恢复为active"
2. 但6.13节【5.标记职位已招满】接口说明中写的是：`"仅当取消后至少一个启用中的工种/级别未满员时，职位才允许恢复为 active"`
3. 两处描述存在微妙差异："未满员工种/级别" vs "启用中的工种/级别未满员"
4. 更严重的是，数据库设计中 `jobs.is_urgent` 与工种/级别的"满员"判断逻辑完全缺失关联定义

**冲突说明**:
- 流程图(5.5)与API说明(6.13)对"恢复条件"的表述不完全一致
- `jobs.is_urgent` 字段的存在与否、置顶/急招状态与"满员"的关联关系未定义

**超规格设计**:
- "职位已招满"的业务定义不清晰：是指所有工种都满员，还是任一工种满员就标记？
- `is_urgent` 字段与 `is_top` 字段的业务边界模糊

**影响**:
- 可能导致前端展示与后端逻辑不一致
- 职位恢复状态的判断缺乏明确标准

**建议修正**:
1. 明确"职位已招满"的业务定义（建议：以工种/级别组合为粒度，任一组合满员即标记）
2. 统一流程图与API描述中的"恢复条件"措辞
3. 定义 `is_urgent` 字段的触发条件和业务意义

**关联修改位置**:
- 3.9 applications表说明
- 5.5 职位报名与审核流程图
- 6.13【5.标记职位已招满】接口说明

## 问题二：用户会话管理中 device_session_id 与端区分冲突

**所在章节**: 3.1.1 用户表(users) / 4.2 登录与注册页面 / 6.1 用户端接口

**原文要点**:
```
users.device_session_id - 设备唯一会话标识(登录时生成)
登录流程(4.2)：多端登录允许，但需要区分当前登录设备
6.1【2.通用认证刷新Token】：根据 device_session_id 标识区分不同登录会话
```

**问题描述**:
1. 文档要求 `device_session_id` 用于区分不同登录会话
2. 但 JWT Token 续期机制(8.2)说的是"按当前 `system_config.token_expire_days` 重新签发新Token"
3. 如果用户换设备登录，原 `device_session_id` 对应的 Token 该如何处理？
4. "多端登录允许"的场景下，旧设备的 Token 是否仍然有效？

**冲突说明**:
- `device_session_id` 的生命周期管理未定义（创建、失效、替换规则）
- JWT Token 的"设备绑定"概念与"多端登录允许"存在矛盾

**超规格设计**:
- "多端登录"的具体业务规则未定义（最多几端？同端多点？）

**影响**:
- Token 安全管理可能存在漏洞
- 用户体验不一致（用户不清楚哪些设备已登录）

**建议修正**:
1. 明确 `device_session_id` 的生命周期管理规范
2. 定义 JWT Token 与 `device_session_id` 的绑定关系
3. 提供用户查看/管理登录设备的接口设计

**关联修改位置**:
- 3.1.1 users表
- 4.2 登录与注册页面
- 6.1 认证接口
- 8.2 Token认证说明
