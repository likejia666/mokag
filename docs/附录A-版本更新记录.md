v2.4 主要更新说明（第七轮问题修订，安全与文件模型收束，2026-03-28）：
1. 【8.1 敏感信息加密】将加密字段收窄为真正敏感业务属性（身份证号），新增【文件访问控制】独立模块，营业执照路径/资质文件路径改为 private disk + 临时签名 URL 机制，不再作为加密字段处理。
2. 【8.1 CSRF防护】改为分场景安全矩阵：用户端/管理端/WebSocket 均为 Bearer Token 鉴权，不适用 CSRF；仅 Cookie Session Web 表单场景才启用 CSRF Token + SameSite + Origin 校验。
3. 【8.2 接口签名】删除泛化口号式描述，改为【高风险操作安全加固】，列明当前安全保障体系（JWT+黑名单+限流+验证码+幂等键+HTTPS），明确高风险操作二次校验要求，预留开放平台签名扩展说明。
4. 【6.10 文件上传】统一上传接口返回结构，新增 file_id、path、disk、mime_type 字段；确立业务表统一存 file_id 或相对 path 的规范，资源输出层统一拼装访问 URL。
5. 【9.3 数据备份】修正 mysqldump 明文密码示例，改为专用备份账号 + ~/.my.cnf 凭据文件方案；补充备份文件压缩、SHA256校验、权限控制、异地同步及恢复演练要求。
6. 补齐企业认证提交流程与管理端详情接口的口径一致性：提交阶段改为"先上传获取 file_id，再校验归属并落 file_id"，文件访问统一通过 private disk + 临时签名 URL 输出，不再使用"路径 AES 加密"或"材料 URL 入库"等旧表述。【更新至v2.4.1】
7. 修正职位关闭状态的业务语义歧义：为 `jobs` 新增 `close_reason` 字段（本次修订新增，v2.4 此前版本文档未定义），用于区分 manual / filled / admin 关闭原因；同时收紧"取消报名后恢复职位"为 `active` 的条件，仅允许在 `status=closed` 且 `close_reason=filled`、并且取消后仍存在未满员启用工种/级别时恢复，并要求恢复时将 `close_reason` 置回 null。

【Laravel 11.x 开发注意事项】
1. 数据库结构变更统一通过 Laravel Migration 管理，禁止直接手改生产表结构后不同步迁移文件
2. 模型定义统一使用 Laravel Eloquent；关联、访问器、作用域、软删除、类型转换等能力需在模型中显式声明
3. 路由统一按 Laravel 路由文件分层管理：API 接口走 routes/api.php，控制台命令与调度走 routes/console.php
4. 中间件、异常处理、请求生命周期装配按 Laravel 11 方式在 bootstrap/app.php 中集中注册和维护
5. 配置管理统一使用 .env + config/*.php，生产环境部署后执行配置缓存，避免运行时直接依赖未缓存配置
6. 缓存、队列、事件、通知、文件存储、日志等基础设施均按 Laravel 组件规范接入，并在部署文档中保持一致
