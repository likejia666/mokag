--------------

【框架变更说明】
- 后端已从 ThinkPHP 6.x 完整迁移为 Laravel 11.x，运行环境要求 PHP 8.2+、MySQL 8.0+、Redis 6.0+
- 应用启动入口与框架装配遵循 Laravel 11 极简骨架：HTTP/Console 路由、异常处理、中间件注册统一通过 bootstrap/app.php 组织
- API 路由以 routes/api.php 为主；命令行任务与调度以 routes/console.php 和 Artisan Command 为主；服务提供者清单位于 bootstrap/providers.php
- 请求校验统一使用 FormRequest 或 Validator；控制器仅负责参数编排、权限校验、事务调度与响应封装，不再沿用 ThinkPHP 控制器/验证器写法
- 数据访问统一使用 Eloquent ORM、Query Builder、DB 事务、Schema Migration、Seeder、Factory 等 Laravel 标准能力，不再使用 ThinkPHP Model/Migrator 语法
- 认证体系统一为 Laravel 11 + JWT Bearer Token（用户端/管理员端通过不同 guard 或中间件区分），接口请求头统一使用 Authorization: Bearer {token}
- 缓存、队列、事件、文件存储、日志、异常处理均按 Laravel 组件约定落地：config/* 配置 + .env 驱动切换 + Artisan 命令管理
- 部署、定时任务、队列消费、WebSocket 常驻进程等运维操作均以 Laravel 11 项目结构为基准，不再沿用 ThinkPHP 的目录和启动方式

【Laravel Migration 说明】
- 迁移文件位于 database/migrations/，数据填充位于 database/seeders/，模型工厂位于 database/factories/
- 使用 `php artisan migrate` 执行迁移，使用 `php artisan migrate:rollback` / `php artisan migrate:fresh --seed` 进行回滚或重建
- 表结构定义统一使用 `Schema`、`Blueprint`、外键约束、索引、唯一索引、软删除字段与时间戳字段；默认主键推荐使用 `$table->id()`
- 业务初始化数据通过 Seeder 管理，测试/演示数据通过 Factory 生成
- 涉及 JSON 字段、全文索引、函数索引等能力时，按 MySQL 8.0+ 特性设计并在迁移中显式声明
- 与 ThinkPHP 迁移差异：
  - Laravel 使用 `Illuminate\Support\Facades\Schema` 与 `Blueprint` 定义结构，不使用 `think\migration\Migrator`
  - Laravel 迁移文件包含 `up()` 与 `down()`，必须保证可回滚
  - Laravel 通过 Seeder / Factory / Artisan 命令串联初始化流程，而不是在迁移中混入大量业务数据写入
  - 生产环境变更应遵循"先迁移、再部署、再执行缓存刷新/队列重启"的 Laravel 发布流程

2.4 实时通讯
------------
技术方案：Workerman + Gateway Worker
通讯协议：WebSocket (ws://)
进程架构：
- Gateway进程：维护客户端连接
- BusinessWorker进程：处理业务逻辑
- Register进程：注册中心协调
心跳机制：客户端30秒心跳包

WebSocket连接与鉴权规范：

【连接地址】
- HTTP环境下：ws://m.mokag.top/ws
- HTTPS环境下：wss://m.mokag.top/ws
- Nginx 对外统一暴露 `/ws` WebSocket 入口，内部反向代理到 GatewayWorker/Workerman 监听端口（如 127.0.0.1:7272）
- 连接时必须携带鉴权 Token（URL 参数方式）：
  ws://m.mokag.top/ws?ws_token={ws_token}
  其中 ws_token 通过 GET /api/ws/config 接口获取

【鉴权流程】
1. 用户登录成功后，调用 GET /api/ws/config 获取 ws_token
2. 建立WebSocket连接时，将 ws_token 作为URL参数携带
3. 服务端验证 ws_token 有效性，验证通过后建立连接
4. 鉴权失败时服务端立即断开连接，返回错误码 9001（WebSocket专用错误码）【修正】【P0】

【WebSocket事件契约总则】【新增】【P0】
- 客户端→服务端消息统一使用 `action`、`data`、`ts`、`request_id` 四个顶层字段；服务端→客户端消息统一使用 `event`、`data`、`ts`、`request_id` 四个顶层字段【新增】【P0】
- 错误事件必须额外返回 `code`、`msg`；若为对某次客户端请求的回执，必须原样带回对应 `request_id`【新增】【P0】
- 事件命名统一使用点分命名：`message.created`、`message.read`、`conversation.updated`、`job.updated`、`application.updated`、`notification.created`、`system.kickout`【新增】【P0】
- 客户端指令统一使用 `action` 表达，例如：`ping`、`message.send`、`message.read`、`join.conversation`、`leave.conversation`、`message.recall`、`message.delete`【新增】【P0】

12. 断线后 HTTP 增量同步（客户端行为说明）
- 重连成功后，客户端必须基于最近游标、`last_msg_id`、`last_notification_id` 等信息通过 HTTP 接口拉取增量数据
- WebSocket 通道仅恢复实时事件流，不承担 `missed_messages` 这类离线补偿专用事件职责
- 客户端需按业务主键、服务端时间与幂等键进行合并，避免重复展示或乱序

【心跳机制】
- 客户端每30秒发送：{"action":"ping","request_id":"req_ping_xxx","ts":1709692800,"data":{}}
- 服务端响应：{"event":"pong","request_id":"req_ping_xxx","ts":1709692800,"data":{"server_time":"2024-03-06 10:00:00"}}
- 连续90秒未收到有效心跳响应，视为断线，触发重连

【断线重连与消息补偿】
1. 客户端检测到断线后，立即尝试重连
2. 重连前先检查当前 ws_token 与 `device_session_id` 是否仍有效；任一失效都必须重新走 HTTP 登录态/GET /api/ws/config 流程
3. 重连成功后，客户端基于最近游标、last_msg_id、last_notification_id 等信息通过 HTTP 接口拉取增量数据
4. 客户端按业务主键与时间顺序做幂等合并，避免重复与乱序
5. WebSocket 通道不再承担 `missed_messages` 这类离线全量补偿职责

【错误事件格式】
{
  "event": "system.error",
  "request_id": "req_xxx",
  "ts": 1709692800,
  "code": 9001,  // WebSocket专用错误码
  "msg": "错误描述",
  "data": {
    "details": {}
  }
}

WebSocket专用错误码：
- 9000: WebSocket连接错误
- 9001: WebSocket Token无效或过期（鉴权失败，仅在WebSocket通道使用，HTTP接口仍使用3001）
- 9002: WebSocket消息格式错误
- 9003: WebSocket发送消息过于频繁
- 9004: WebSocket消息内容违规
- 9005: WebSocket对方已拉黑你
- 9006: WebSocket对方不在线（仅用于特定场景）
- 9007: WebSocket会话不存在或已关闭
- 9008: WebSocket消息撤回超时（超过2分钟）
- 9009: WebSocket无权执行此操作

【鉴权失败处理】
- 握手阶段鉴权失败：服务端返回 `system.error`（code=9001）或按网关能力直接拒绝连接
- 已建立连接后若发现 `device_session_id` 失效：服务端必须先发送 `system.kickout`，再主动断开连接
- 前端收到 `system.error` / `system.kickout` 后，必须立即清理本地登录态与连接状态，并重新走 HTTP 登录/刷新配置流程

2.5 文件存储
------------
存储方式：本地服务器存储（Laravel Filesystem 本地磁盘）
上传限制：
- 单张图片 ≤ 5MB
- 最多3张图片
- 支持格式：jpg, jpeg, png, gif
处理流程：
1. 自动压缩
2. 生成缩略图
3. 添加平台水印（右下角，水印内容为平台名称+域名，透明度30%，字体14px白色）

【Laravel 11 文件存储规范】
- 统一通过 Laravel Filesystem 操作文件，默认使用 `storage/app/public` 磁盘，并通过 `php artisan storage:link` 暴露到 `public/storage`
- 业务表默认保存 `file_id` 或受控的相对 path；对外接口统一输出 File Resource 对象，不得直接向前端暴露裸路径或长期有效 URL，也不建议在业务接口中长期保存原始 base64 内容
- 上传入口统一使用 `multipart/form-data`；控制器接收 `UploadedFile` 对象，交由 FormRequest 校验后再进入文件服务类处理
- 图片压缩、缩略图、水印、违规审核等流程建议封装为独立 Service/Job，避免控制器内堆积文件处理逻辑
- 涉及身份证、营业执照等敏感资质文件时，应区分"公开访问文件"与"受控访问文件"：敏感文件默认存储在非公开目录，通过授权下载接口或临时签名 URL 访问
- 【敏感文件访问控制规范】
  - 访问方式：通过授权下载接口获取临时签名URL
  - 签名URL有效期：15分钟（可配置）
  - 生成方式：由服务端按需生成，Laravel实现为 Storage::disk('private')->temporaryUrl($path, $expiry)
  - 权限校验：验证用户身份及文件访问权限后才返回签名URL
  - 禁止在任何接口响应中直接返回私有文件的真实存储路径
- 目录建议按业务域划分：`companies/`、`resumes/`、`jobs/`、`reports/`；文件名使用 UUID/哈希值生成，避免原始文件名泄露
- 若后续迁移到 OSS / COS / S3，仅需切换 Filesystem 磁盘配置，不改业务接口协议

2.6 域名与服务器
----------------
域名：m.mokag.top
服务器：需支持PHP 8.2、MySQL 8.0+、WebSocket、Redis

【PHP 8.2 环境要求】
- PHP版本：>= 8.2.0
- 必需扩展：pdo_mysql, mbstring, openssl, json, fileinfo, ctype, xml, bcmath, tokenizer
- 可选扩展：redis, swoole (用于高性能WebSocket)
- Web服务器：Nginx 1.20+ 或 Apache 2.4+ (需配置URL重写)
- 数据库：MySQL 8.0+ 或 MariaDB 10.6+
- 缓存：Redis 6.0+ (推荐，用于会话、缓存、队列)

【Laravel 11.x 特有配置】
- 应用目录结构遵循 Laravel 11 标准：app/、bootstrap/、config/、database/、public/、resources/、routes/、storage/
- 应用装配入口为 bootstrap/app.php；服务提供者清单维护于 bootstrap/providers.php
- 环境配置统一使用 .env 与 config/*.php，支持开发/测试/生产多环境切换与配置缓存
- API 路由位于 routes/api.php；控制台命令与调度定义位于 routes/console.php；广播、事件、队列、异常等能力按 Laravel 11 约定接入
- 队列驱动支持：database / redis / sqs；缓存驱动支持：file / redis / memcached；会话驱动支持：file / redis / database
- 生产环境建议启用 `php artisan config:cache`、`php artisan route:cache`、`php artisan view:cache`、`php artisan event:cache` 等优化命令

--------------------
格式：JOB + YYYYMMDD + 6位序号
