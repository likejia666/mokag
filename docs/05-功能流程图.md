5.1 用户注册流程
--------------------------------------------------
开始
  ↓
用户选择注册类型（求职者/雇主）
  ↓
输入手机号、密码、确认密码
  ↓
输入图形验证码
  ↓
勾选用户协议
  ↓
点击注册按钮
  ↓
<校验>
├─ 手机号格式是否正确？ ──否→ 提示错误，返回修改
├─ 手机号是否已注册？ ──是→ 提示已注册，跳转登录
├─ 密码是否符合规则？ ──否→ 提示错误，返回修改
├─ 两次密码是否一致？ ──否→ 提示不一致，返回修改
├─ 验证码是否正确？ ──否→ 提示错误，刷新验证码
└─ 是否勾选协议？ ──否→ 提示勾选协议
  ↓
<全部通过>
  ↓
生成用户记录（users表）
  ↓
设置用户角色（role字段）
  ↓
记录注册IP和设备ID
  ↓
自动登录
  ↓
<判断角色>
├─ 求职者 → 跳转完善简历页
└─ 雇主 → 跳转企业认证页
  ↓
结束


5.2 用户登录流程（含登录锁定）
--------------------------------------------------
开始
  ↓
用户输入手机号、密码
  ↓
输入图形验证码
  ↓
点击登录按钮
  ↓
<校验>
├─ 手机号格式是否正确？ ──否→ 提示错误
├─ 验证码是否正确？ ──否→ 提示错误，刷新验证码
└─ 手机号是否存在？ ──否→ 提示未注册，跳转注册
  ↓
<检查账号锁定状态>
├─ login_locked_until > 当前时间？ ──是→ 提示账号已锁定，显示剩余锁定时间
  ↓
<验证密码>
├─ 密码是否正确？
│   ├─ 否→ login_fail_count + 1
│   │       ├─ fail_count >= system_config.login_fail_max → 锁定账号 system_config.login_lock_minutes 分钟，更新login_locked_until
│   │       │                     提示：密码错误次数过多，账号已被临时锁定
│   │       └─ fail_count < system_config.login_fail_max  → 提示：密码错误，还剩X次机会
  ↓
<检查账号状态>
├─ 是否在黑名单？ ──是→ 提示账号被封禁，拒绝登录
├─ 是否已注销？ ──是→ 提示账号已注销
  ↓
<登录成功>
  ↓
重置 login_fail_count = 0，清空 login_locked_until
  ↓
生成登录Token
  ↓
记录登录日志（operation_logs表）
  ↓
更新最后登录时间和IP
  ↓
跳转首页
  ↓
结束

5.3 职位发布流程（含草稿）
--------------------------------------------------
开始（雇主端）
  ↓
点击【发布职位】按钮
  ↓
<检查登录状态>
├─ 未登录 → 跳转登录页
  ↓
<检查认证状态>
├─ 未提交认证/已拒绝 → 提示完成企业认证（跳转认证页）
├─ 审核中 → 允许进入发布页并保存草稿，但不可正式发布
└─ 已认证 → 进入发布页面
  ↓
<是否有草稿？>
├─ 是 → 提示"发现未完成的草稿，是否继续编辑？"
│   ├─ 是 → 加载草稿内容
│   └─ 否 → 新建职位
  ↓
填写职位信息
  ↓
<用户点击【保存草稿】或自动保存>
  ↓
<jobs表插入或更新>
├─ status = draft（草稿）
└─ 不做必填项校验
  ↓
提示草稿已保存
  ↓
<用户点击【预览并发布】>
  ↓
<校验必填项>
├─ 职位标题 ──未填→ 提示必填
├─ 工作地点 ──未填→ 提示必填
├─ 至少一个工种且至少一个级别已勾选 ──否→ 提示
├─ 已勾选级别的工价和人数 ──未填或≤0→ 提示
└─ 联系方式 ──未填→ 提示必填
  ↓
<校验企业认证状态>
├─ 非已认证 → 拦截正式发布，提示仅可保存草稿
└─ 已认证 → 跳转预览页
  ↓
用户确认发布
  ↓
生成职位编号（job_no）
  ↓
<插入/更新jobs表>
  ↓
<插入job_levels表（仅is_enabled=1的级别）>
  ↓
<判断职位是否需审核>
├─ 命中职位审核规则（如违规词/敏感内容/图片异常/联系方式异常/风控策略）→ status = pending，audit_status = 0 → 提交职位审核队列
├─ 企业认证时间超过30天 → status = active，audit_status = 1（直接发布，视为系统自动审核通过）
├─ 企业认证时间≤30天：
│  ├─ 统计该企业认证通过后发布的职位数量（status != 'draft'）【明确计数规则】
│  │   ├─ 排序规则：按 created_at ASC（按发布时间升序，取最早发布的职位）
│  │   ├─ 统计范围：包括所有非草稿状态的职位（active/paused/closed），草稿不计入
│  │   ├─ 不包括：已删除的职位
│  │   └─ 说明："前N条" = 按发布时间最早发布的N条职位，这些需要审核
│  ├─ 数量 < first_jobs_audit_limit（默认10） → status = pending，audit_status = 0（待审核）
│  └─ 数量 >= first_jobs_audit_limit → status = active，audit_status = 1（直接发布，视为系统自动审核通过）
│  └─ 配置项 first_jobs_audit_limit 可在管理员后台系统配置中调整
  ↓
发布成功 → 跳转职位管理页
  ↓
结束

5.4 职位报名流程
--------------------------------------------------
开始（求职者端）
  ↓
浏览职位列表
  ↓
点击职位卡片
  ↓
进入职位详情页
  ↓
查看完整职位信息
├─ 所有可选工种
├─ 每个工种的三个级别
└─ 各级别工价和名额
  ↓
点击【立即报名】按钮
  ↓
<检查登录状态>
├─ 未登录 → 跳转登录页
  ↓
<检查简历状态>
├─ 未完善简历 → 跳转简历编辑页，提示完善简历
  ↓
<基础资格检查>
├─ 职位状态是否为招聘中？ ──否→ 提示职位已关闭
├─ 用户是否被限制报名？ ──是→ 提示限制原因
├─ 被拒绝且在限制期内？ ──是→ 提示X小时后可重新报名
├─ 同手机号/同设备是否触发风控规则？ ──是→ 标记风控或转人工审核，不覆盖业务唯一性
  ↓
<弹出报名窗口>
  ↓
显示所有可选工种列表
  ↓
用户选择一个工种（单选）
  ↓
显示该工种的三个级别：
├─ 大工（工价、剩余名额）
├─ 中工（工价、剩余名额）
└─ 小工（工价、剩余名额）
  ↓
用户选择一个级别（单选）
  ↓
<检查该工种级别报名资格>
├─ 当前用户在该职位该工种该级别是否已有有效报名（is_latest=1 且状态有效）？ ──是→ 提示已报名，显示报名状态
├─ 该级别剩余名额 > 0？ ──否→ 提示名额已满，请选择其他级别
  ↓
填写留言（可选）
  ↓
点击【确认报名】
  ↓
<提交报名>
  ↓
<事务写入applications表>
├─ 先将该用户该职位该工种该级别旧记录 is_latest 置0
├─ job_id（职位ID）
├─ user_id（用户ID）
├─ job_level_id（职位级别明细ID）
├─ category_id（工种ID）
├─ level（大工/中工/小工）
├─ message（留言）
├─ status（applied已报名，前端展示文案为"待审核"）
├─ is_latest = 1
├─ device_id（设备标识，用于风控）
└─ created_at（报名时间）
  ↓
<更新job_levels表>
├─ 找到对应的工种级别记录
└─ applied_count（已报名人数）+ 1
  ↓
<检查是否招满>
├─ 当前级别 applied_count >= recruit_count？ ──是→ 该级别停止继续报名
├─ 所有启用的工种/级别是否均已满员？
│   ↓
│ 是 → 更新jobs表：status = closed，close_reason = filled（已关闭，原因：已招满）
  ↓
<发送通知>
├─ 给求职者：报名成功，等待审核
└─ 给雇主：收到新报名（推送/站内信）
  ↓
报名成功
  ↓
显示报名成功提示
  ↓
跳转我的报名页
  ↓
结束

5.6 取消报名流程
--------------------------------------------------
开始（求职者端）
  ↓
进入【我的报名】页面
  ↓
查看报名记录列表
  ↓
找到要取消的报名
  ↓
<检查报名状态>
├─ status = applied（前端展示为"待审核"）→ 可以取消
├─ status = accepted（已通过）→ 可以取消
└─ status = rejected（已拒绝）→ 无法取消（已结束）
  ↓
点击【取消报名】按钮
  ↓
<弹出确认窗口>
├─ 提示：确定要取消报名吗？
└─ 说明：取消后名额将恢复
  ↓
用户确认取消
  ↓
<更新applications表>
├─ status = cancelled（已取消）
├─ is_latest = 0（取消后不再是当前有效报名）
└─ updated_at = 当前时间
  ↓
<更新job_levels表>
├─ 找到对应工种级别记录
└─ applied_count（已报名人数）- 1
  ↓
<系统自动检查职位状态>
├─ 触发方式：【系统自动恢复】（在取消报名同一事务中同步处理，无需人工干预）
├─ 恢复条件判断：
│   ├─ 职位当前 status = closed 且 close_reason = filled（已关闭，原因：已招满）
│   └─ 且取消后至少一个启用中的工种/级别未满员
├─ 满足条件 → 自动恢复
│   ├─ 将职位 status 恢复为 active
│   └─ 将 close_reason 置为 null
└─ 不满足条件（close_reason = manual / admin）→ 不恢复，保持关闭状态
  ↓
<发送通知>
├─ 给求职者：报名已取消
└─ 给雇主：有求职者取消了报名
  ↓
取消成功
  ↓
刷新报名列表
  ↓
结束

├─ 历史消息记录

  ↓
<发送消息>
  ↓
用户输入消息内容
  ↓
点击发送或选择表情
  ↓
<内容检测>
├─ 是否包含违禁词？ ──是→ 拦截并提示
├─ 是否包含联系方式？ ──是（如果开启过滤）→ 拦截并提示
├─ 是否重复内容刷屏？ ──是→ 拦截并提示
  ↓
<通过检测>
  ↓
<插入messages_private表>
├─ conversation_id（关联会话）
├─ from_user_id（发送者）
├─ to_user_id（接收者）
├─ job_id（关联职位）
├─ client_msg_id（客户端幂等消息ID）
├─ content（消息内容）
├─ message_type（文本/表情）
├─ is_read = 0（未读）
├─ created_at（发送时间）
└─ expire_at = 当前时间 + system_config.message_retention_days（默认30天，由消息服务统一写入）
  ↓
<WebSocket实时推送>
├─ 推送给接收者
└─ 更新未读消息数
  ↓
消息发送成功
  ↓
显示在聊天界面
  ↓
<接收者查看消息>
  ↓
进入聊天界面
  ↓
<更新消息状态>
└─ is_read = 1（已读）
  ↓
<定时清理>
├─ 系统定时任务
└─ 删除expire_at < 当前时间的消息
  ↓
结束

<发起举报>
├─ 在职位详情页点击【举报】
├─ 在评论下方点击【举报】
└─ 在聊天界面点击【举报用户】
  ↓
<弹出举报窗口>
  ↓
选择举报类型：
├─ 虚假职位
├─ 诈骗信息
├─ 辱骂骚扰
├─ 违规图片
└─ 其他
  ↓
填写详细描述
  ↓
上传证据（可选，图片）
  ↓
点击【提交举报】
  ↓
<插入reports表>
├─ reporter_id（举报人）
├─ report_type（举报类型）
├─ target_id（被举报对象ID）
├─ reason（举报原因）
├─ description（详细描述）
├─ evidence（证据文件ID数组，输出层统一转换为 evidence_files）
├─ status = pending（待处理）
└─ created_at（举报时间）
  ↓
提示：举报已提交，我们会尽快处理
  ↓
--------------------------------
开始（管理员端）
  ↓
登录管理员后台
  ↓
进入【举报处理】模块
  ↓
查看待处理举报列表
  ↓
点击查看举报详情
├─ 举报类型
├─ 被举报对象信息
├─ 举报原因和描述

5.22 WebSocket连接建立与断线重连流程【新增】

开始
  ↓

  ↓
<获取WebSocket连接配置>
└─ 调用 GET /api/ws/config 获取 ws_url / wss_url 与 ws_token
  ↓
<建立WebSocket连接（URL参数鉴权）>
├─ 根据当前环境选择协议：
│   ├─ HTTP环境 → ws://m.mokag.top/ws?ws_token={ws_token}
│   └─ HTTPS环境 → wss://m.mokag.top/ws?ws_token={ws_token}
└─ ws = new WebSocket((isHttps ? wss_url : ws_url) + '?ws_token=' + ws_token)
  ↓
<连接建立>
├─ 触发 onopen 事件
└─ 服务端自动验证URL中的token
  ↓
<鉴权结果>
├─ 鉴权成功 → 连接建立完成，进入正常事件收发
│   ↓
│ <开始心跳>
│ ├─ 每30秒发送心跳包：{"action":"ping","request_id":"req_ping_xxx","ts":1709692800,"data":{}}
│ └─ 服务端响应：{"event":"pong","request_id":"req_ping_xxx","ts":1709692800,"data":{}}
│   ↓
│ <正常通信>
│ ├─ 接收事件 → 触发 onmessage 事件
│ ├─ 发送指令 → 调用 ws.send()
│ └─ 处理各类事件（私聊/通知/在线人数/强制下线等）
│
└─ 鉴权失败 → 服务端返回 system.error 或直接断开连接（code=9001）【修正】【P0】
    ↓
    前端提示用户重新登录并清理本地连接状态
  ↓
<检测到断线>
├─ 触发 onclose 或 onerror 事件
└─ 连续90秒未收到有效心跳或 `pong` 响应
  ↓
<UI状态提示>
└─ 顶部显示黄色提示条："网络连接不稳定，正在尝试重连...（1/10）"
  ↓
<断线重连策略（指数退避）>
├─ 第1次重连：等待1秒后重连
├─ 第2次重连：等待2秒后重连
├─ 第3次重连：等待4秒后重连
├─ 第4次重连：等待8秒后重连
├─ 第5次重连：等待16秒后重连
├─ 第6次及以后：等待30秒（最大值）
└─ 重连次数上限：10次
  ↓
<重连前检查 ws_token 与 device_session_id>
├─ 已收到 `system.kickout` 或本地确认 `device_session_id` 失效 → 停止自动重连，清理本地登录态与连接状态，重新走 HTTP 登录/刷新配置流程
├─ `ws_token` 未过期且 `device_session_id` 仍有效 → 直接携带原 `ws_token` 发起重连
└─ `ws_token` 已过期或无法确认 `device_session_id` 有效性 → 重新调用 GET /api/ws/config 获取新 `ws_token` 后再重连
  ↓
<重连成功>
├─ 重置重连计数器
├─ 隐藏提示条
├─ 恢复心跳机制
├─ 基于最近游标、`last_msg_id`、`last_notification_id` 通过 HTTP 接口拉取增量数据
└─ 完成增量合并后恢复正常通信
  ↓
<重连失败（达到上限10次）>
├─ 停止重连
├─ 更新提示条："网络连接已断开，请检查网络后刷新页面"
├─ 显示【刷新页面】按钮
└─ 用户手动点击后重新建立连接
  ↓
结束

说明：
- 统一使用URL参数方式携带token进行鉴权（不再支持首条消息鉴权）
- ws_token有效期较短（建议5分钟），过期后需重新调用 GET /api/ws/config 获取新的 ws_token 再建立连接；ws_token 仅用于握手鉴权，与登录 token 独立管理
- 重连时不能只判断 `ws_token` 是否过期，还必须同时校验对应 `device_session_id` 是否仍有效；即使 `ws_token` 未过期，只要 `device_session_id` 已失效，也不得继续自动重连
- 若客户端已收到 `system.kickout`，或确认当前设备登录态已失效，必须立即停止自动重连并重新走 HTTP 登录/刷新配置流程
- 断线后的历史消息、通知补偿统一通过 HTTP 接口拉取，WebSocket 连接恢复后仅继续承担实时事件流与回执职责
- **主动关闭**：用户退出登录或切换页面时，客户端应发送可选的 leave.conversation 事件后调用 ws.close() 正常关闭连接，不触发重连
- **异常断开**：网络故障/心跳超时导致连接断开，客户端检测到 onclose/onerror 事件后自动按重连策略重试

5.24 Token自动续期流程【新增】【P1】
--------------------------------------------------
开始
  ↓
用户发起API请求
  ↓
<服务端验证Token>
├─ Token无效/过期 → 返回3001/3002错误，前端跳转登录
  ↓
<检查Token有效期>
├─ 剩余有效期 > 1天 → 正常处理请求
  ↓
<Token即将过期>
├─ 剩余有效期 ≤ 1天
  ↓
<生成新Token>
├─ 基于当前用户信息生成新Token
├─ 新Token有效期 = 当前时间 + system_config.token_expire_days（默认7天）
  ↓
<返回响应>
├─ 正常响应数据
├─ 响应头 X-New-Token: {新Token}
  ↓
<前端处理>
├─ 检测到X-New-Token头
├─ 更新本地存储的Token
  ↓
后续请求使用新Token
  ↓
结束

说明：
- Token有效期由 system_config.token_expire_days 控制（默认7天），到期前1天内自动续期
- 修改 token_expire_days 仅影响后续新签发/续签的 Token，已签发 Token 保持原过期时间
- 每次请求都可能获得新Token
- 前端需监听X-New-Token头并更新
- 客户端收到 `X-New-Token` 后，必须立即以新 token 覆盖本地旧 token
- 为避免并发请求抖动，旧 token 可保留最多 60 秒宽限期；宽限期仅用于已在途请求与短时并发，不应用于长期并存
- 宽限期结束后旧 token 必须失效；若旧 token 已被加入 Redis 黑名单，则无论是否仍在宽限期内均立即失效
- 新旧 token 必须归属于同一 `device_session_id`，自动续期不会创建新的设备会话，也不会改变"当前设备 / 其他设备"的归属关系【新增】【P1】
- 若账号已注销、被封禁、被管理员强制下线，或该 `device_session_id` 已失效，则不得继续自动续期，应直接返回认证失败并要求重新登录【新增】【P0】
- 【并发请求处理】服务端在宽限期内只对"携带即将过期Token"的第一个请求返回X-New-Token；后续请求正常处理但不返回新Token；避免客户端收到多个不同新Token造成混乱【新增】【P0】
