【Laravel 11 数据层实现约定】
- 所有表结构由 Migration 管理；主键默认使用 bigint unsigned 自增（Laravel `$table->id()`）
- 外键字段统一使用 `foreignId()` / `foreignIdFor()` 定义，并结合 `constrained()`、`cascadeOnDelete()`、`nullOnDelete()` 明确约束策略
- 时间字段优先使用 `$table->timestamps()`；软删除表增加 `$table->softDeletes()`，并在模型中使用 `SoftDeletes`
- 枚举类字段在数据库层可使用 ENUM/TINYINT，在 Laravel 模型层建议通过 PHP 8.2 Enum 或常量映射统一语义
- JSON 扩展字段优先使用 JSON 类型，并在 Eloquent 模型中通过 `$casts` 声明 `array`/`json`
- 涉及计数缓存字段（如 unread_message_count、views_count、comments_count）需明确"最终以数据库聚合校准"为准，异步更新仅用于性能优化
- 涉及敏感字段需通过 cast、accessor、service 或加密组件统一处理，不在控制器中分散处理：
  * 加密存储字段（AES-256）：身份证号、法人身份证号、银行卡号等真正敏感业务属性
  * 访问控制字段：营业执照路径、资质文件路径、证件图片路径等文件定位信息，采用 private disk + 临时签名 URL 机制保护，不做字段级加密
  * 脱敏展示字段：手机号（输出时掩码处理，如 138****0000）
- 建议按业务域建立 Model、Repository/QueryObject（可选）、Service、Policy、Observer，避免把复杂查询和状态机直接堆在控制器

--------------------------------------------------------------------------------

is_certified          TINYINT           是否认证：0未认证/1审核中/2已认证/3认证被拒绝
is_blacklist          TINYINT           是否黑名单：0否/1是
is_deleted            TINYINT           是否注销：0否/1申请中/2已注销（对应账号状态：active / pending_deletion / deleted；封禁状态由 is_blacklist 共同决定，is_blacklist=1 时账号状态视为 disabled）【更新】【P0】
delete_apply_time     DATETIME          注销申请时间

unread_message_count  INT               未读消息数缓存（含私聊+通知）【新增】
must_change_password  TINYINT           是否强制修改密码：0否/1是【新增】
created_at            DATETIME          创建时间
updated_at            DATETIME          更新时间
索引：
- PRIMARY KEY (id)
- UNIQUE KEY (phone)
- INDEX (role, is_certified)
- INDEX (is_blacklist)
- INDEX (must_change_password)【新增】

说明：
- login_fail_count：连续登录失败达到 system_config.login_fail_max 后触发锁定，锁定时长由 system_config.login_lock_minutes 控制
- login_locked_until：锁定到期后自动清零 login_fail_count
- 【账号状态优先级】（从高到低）：1)已注销(is_deleted=2) 2)账号锁定(login_locked_until>now()) 3)黑名单(is_blacklist=1) 4)注销申请中(is_deleted=1) 5)正常。互斥规则：is_deleted=2时is_blacklist必须为0
- unread_message_count更新时机：
  1. WebSocket收到新消息时实时+1
  2. 用户标记消息已读时-1
  3. 用户退出登录后，下次登录时从数据库重新计算同步
  4. 定时任务每小时校准一次（防止计数不准）
  更新方式：异步更新，不阻塞主流程
- must_change_password：管理员为用户修改密码后可标记为1，用户首次登录强制修改密码

【核心业务状态机总则】【新增】【P0】
- 用户账号状态统一定义为：active正常 / pending_deletion注销申请中 / disabled已封禁 / deleted已注销
- 企业认证状态统一定义为：not_submitted未提交 / pending审核中 / approved已通过 / rejected已拒绝；已通过后允许重新提交材料，状态重新进入 pending
- 职位状态统一定义为：draft草稿 / pending待审核 / rejected审核拒绝 / active招聘中 / paused已暂停 / expired已过期 / closed已关闭 / deleted已删除
- 报名状态统一定义为：applied已报名 / cancelled已取消 / accepted已通过 / rejected已拒绝
- 置顶申请状态统一定义为：pending待审核 / approved已通过 / rejected已拒绝 / expired已过期 / cancelled已取消
- 非法状态流转必须返回"状态不允许此操作（4003）"或更具体的业务错误码，禁止由控制器或前端自行推断并绕过状态校验【新增】【P0】
- 需要历史追踪的对象（报名、企业认证、置顶申请）应保留历史记录或快照，不通过覆盖旧状态丢失审计信息【新增】【P1】

│   ├─ 触发方式：【系统自动恢复】（无需人工干预，在取消报名同一事务中处理）
│   ├─ 恢复条件：
│   │   ├─ 职位当前 status = closed
│   │   ├─ close_reason = filled（已招满关闭）
│   │   └─ 取消后至少一个启用中的 job_levels 未满员
│   ├─ 恢复操作：status 恢复为 active，close_reason 置回 null

  - report：关联 reports 表，JOIN reports ON notifications.related_id = reports.id
  - message：关联 messages_private 表，JOIN messages_private ON notifications.related_id = messages_private.id

- Laravel 实现建议：若采用 Eloquent 多态，可将 `related_type` 统一映射为 morph 类型别名，并在模型中通过 `morphTo()` / `morphMany()` 管理；若出于历史兼容继续使用业务字符串，也应集中封装解析逻辑，避免散落在各控制器中

- UNIQUE KEY (user_id, device_id)：同一用户在同一物理设备重复登录时覆盖原会话记录，device_id用于设备管理列表展示

- 设备列表API响应中使用device_id展示给用户，内部管理（踢出连接、自动续期）围绕device_session_id进行【新增】
- session_token 存储Token哈希，用于验证、识别当前请求会话和撤销
- 当前设备由当前请求携带的 token/session_token 动态判断，不在表中持久化存储 is_current
- is_revoked 用于标记用户主动退出或被移除的设备
- 定时任务清理过期（expire_at < 当前时间）且已撤销的会话记录
