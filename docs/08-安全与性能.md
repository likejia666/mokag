8.1 数据安全
--------------------------------------------------

【敏感信息加密】
- 使用AES-256加密算法
- 加密字段（仅限真正的敏感业务属性）：
  * 身份证号
  * 法人身份证号（legal_person_id_card）
  * 银行卡号（如后续涉及结算功能）
- 加密密钥存储于环境变量（APP_ENCRYPT_KEY），不写入代码仓库
- 后台查看需权限验证（admin:cert:view 权限）
- Laravel 实现建议：使用 Eloquent $casts 配合自定义 EncryptedCast，或使用 encrypt()/decrypt() 辅助函数，在 Model 层透明处理

说明：
- 营业执照路径、资质文件路径属于文件存储定位信息，不适合作为加密字段
  原因：路径字段需参与文件访问、展示、后台审核、缩略图生成、CDN迁移、对象存储回收等流程，加密后链路无法正常运作
- 文件路径/file_id 的安全保护应通过以下访问控制机制实现（见下方【文件访问控制】）

【文件访问控制】
- 公开文件（职位图片、用户头像）：存储于 public disk，可直接访问或经 CDN 分发
- 私有文件（营业执照、资质证件、身份证照片）：存储于 private disk（私有桶），数据库只存相对路径或 file_id
- 私有文件访问方式：服务端按需生成临时签名 URL（有效期建议15分钟），通过接口返回给有权限的调用方
  * Laravel 实现：Storage::disk('private')->temporaryUrl($path, now()->addMinutes(15))
  * 或对接对象存储（OSS/COS/S3）的私有桶 + 预签名 URL 机制
- 后台审核查看资质文件：须校验管理员具备 admin:cert:view 权限，再返回签名 URL，不直接暴露存储路径
- 禁止在任何接口响应中直接返回私有文件的真实存储路径或 bucket key

【密码安全】
- 使用bcrypt哈希算法
- 加盐处理
- 最小长度6位，最大20位
- 支持字母、数字、特殊字符

【XSS防护】
- 所有用户输入进行HTML转义
- 富文本编辑器使用白名单过滤
- CSP内容安全策略

【SQL注入防护】
- 使用预处理语句
- 参数化查询
- ORM框架保护

【CSRF防护】

说明：本系统接口体系以 JWT Bearer Token 为主鉴权机制，CSRF 的适用范围需按场景区分，避免对纯 API 接口引入不必要或有副作用的校验。

场景一：用户端 / 雇主端 API（uniapp App + H5）
- 鉴权方式：Authorization: Bearer {token}（JWT）
- CSRF 适用性：不适用
  原因：Bearer Token 通过 Authorization Header 传递，浏览器跨站请求不会自动携带该 Header，天然免疫 CSRF
- 安全措施：JWT 签名校验 + Token 黑名单 + 请求频率限制 + HTTPS

场景二：管理端 API（Admin Bearer Token）
- 鉴权方式：Authorization: Bearer {admin_token}（独立 guard）
- CSRF 适用性：不适用（同上，Bearer Header 鉴权）
- 安全措施：管理员 JWT + 权限中间件（CheckAdminPermission）+ IP 白名单（可选）+ 操作审计日志

场景三：WebSocket 连接
- 鉴权方式：握手阶段 URL 参数传递 ws_token（通过 GET /api/ws/config 获取）
- CSRF 适用性：不适用
- 安全措施：ws_token 短有效期（5分钟）+ 握手后绑定连接身份 + 使用后立即失效

场景四：Cookie Session Web 表单（如后续引入传统 Web 后台）
- 鉴权方式：Laravel Session + Cookie
- CSRF 适用性：必须启用
- 安全措施：
  * Laravel CSRF Token（VerifyCsrfToken 中间件）
  * SameSite=Lax 或 Strict Cookie 属性
  * Origin / Referer Header 校验
  * 当前版本管理端已采用 Bearer Token，本场景暂不适用，留存备用

8.2 接口安全
--------------------------------------------------

【Token认证】
- 采用 JWT Bearer Token 方案，适用于 uniapp App/H5 接口鉴权；请求头统一使用 `Authorization: Bearer {token}`
- 有效期：读取 `system_config.token_expire_days`（默认7天）
- 自动续期机制：距到期不足1天时，每次请求按当前 `system_config.token_expire_days` 重新签发新Token（响应头返回 `X-New-Token`）
- 配置变更规则：修改 `token_expire_days` 仅影响后续新签发/续签的 Token，已签发 Token 保持原过期时间
- 支持主动退出失效：服务端维护 Token 黑名单（Redis），失效 Token 保留至原过期时间
- Laravel 实现建议：通过自定义认证 guard / 中间件解析 JWT，区分用户端、管理员端与 WebSocket 临时令牌，不与 Sanctum Session 模式混用
- 建议为用户端、管理员端、WebSocket 临时令牌分别定义不同的 claims / token_type，并在中间件中强校验，避免跨端 token 误用

【请求频率限制】
- 登录接口：5次/分钟
- 注册接口：3次/分钟
- 获取验证码：3次/分钟（同IP）
- 发送消息：20次/分钟
- 发布职位：10次/小时
- 其他接口：60次/分钟
- 限流统一规则：所有触发限流的 HTTP 接口必须返回 HTTP 429、业务 `code`=6000-6999，并在响应头返回 `Retry-After`，响应体返回 `retry_after`（秒）
- WebSocket 写操作若命中限流，必须返回 `system.error` 事件，包含 `code`、`msg`、`request_id`、`retry_after`，不得静默丢弃
- Laravel 实现要求：在 `RouteServiceProvider` 或专用限流配置中使用 `RateLimiter::for()` 按用户ID、IP、设备标识、接口类型、资源标识组合限流；未登录场景至少按 IP + 设备标识限流

【IP黑名单】
- 自动识别恶意IP
- 永久或临时封禁
- 管理员手动管理

【高风险操作安全加固】

说明：原「接口签名」条目表述过于泛化（重要接口增加签名验证/防止参数篡改/时间戳防重放），缺乏适用范围、签名算法、密钥管理等可执行细节，在当前 JWT Bearer 体系下无法直接落地，故按以下方式重新表述。

当前阶段安全保障体系（已覆盖）：
- JWT Bearer Token 签名校验（防伪造）
- Token 黑名单（Redis，防重用）
- 请求频率限制（防暴力/滥用）
- 图形验证码（登录/注册/敏感操作）
- 幂等键（client_msg_id，防重放）
- HTTPS 全链路加密（防中间人/篡改）

高风险操作二次校验要求：
以下操作在 JWT 鉴权基础上须额外增加身份二次确认：
- 修改手机号 / 邮箱：发送短信/邮件验证码，验证后方可变更
- 修改登录密码：需验证原密码
- 管理员执行封号/解封、删除、批量审核、批量删除、权限变更、系统配置修改、敏感数据导出：必须二次确认，并填写操作原因

【后台高危操作审计与合规总则】
- 后台所有高危写操作必须写入 `operation_logs`，且成功、失败、拒绝三类结果都必须留痕，不得只记录成功操作
- 审计字段至少包括：`operator_id`、`operator_type`、`operation_type`、`operation_module`、`target_id`、`target_type`、`request_id`、`result`、`risk_level`、`reason`、`ip_address`、`user_agent`、`before_snapshot`、`after_snapshot`、`created_at`
- 高危操作范围至少包括：封号/解封、删除、审核、批量操作、权限变更、系统配置修改、数据导出
- `risk_level` 统一使用 `low/medium/high/critical`；其中权限变更、系统配置修改、删除用户、敏感数据导出至少标记为 `high` 或 `critical`
- 高危操作必须支持按 `request_id` 串联一次请求内的主操作、批量明细、异步任务和最终结果，保证可追溯
- `before_snapshot`、`after_snapshot` 仅记录审计所需摘要，不得无选择地落完整敏感原文；密码、token、身份证号、银行卡号、完整手机号、完整邮箱等必须脱敏或仅记录字段名
- 审计日志仅允许查看与按保留期定时清理，不允许业务侧手动修改或提前删除
- 配置变更日志、权限变更日志、导出审计日志的保留期不得低于普通操作日志保留期
- 导出操作除记录 `export_tasks` 外，必须同步写审计日志，记录导出类型、筛选条件、字段范围、数据范围、记录数、发起人、任务状态、文件过期时间与下载时间
- 导出文件必须限时下载，过期自动失效；下载行为本身也必须补充审计记录或更新下载审计时间
- 操作日志查询必须支持按操作者、模块、目标对象、风险等级、时间范围、结果状态筛选，并支持按目标对象或 `request_id` 追溯完整操作链
- Laravel 实现要求：高危后台接口统一通过 FormRequest + Policy/Gate + Service/Action + 审计记录服务落地，禁止在控制器中零散拼装审计字段
- 如命中数据范围限制、敏感字段导出限制或高危权限不足，必须返回 403，并写入失败审计日志说明拒绝原因
- 批量操作必须为每次提交生成批次号或 `request_id`，并能关联到每条明细结果，便于审计、补偿与追责
- 系统配置修改除写入 `config_change_logs` 外，必须同时写入 `operation_logs`，确保配置变更与后台操作审计口径一致
- 子管理员权限变更必须记录授权前后权限差异摘要，且仅允许具备对应高权限的管理员执行
- 敏感数据导出默认遵循最小字段原则；若需导出高敏字段，必须具备额外权限并在审计日志中明确记录原因
- 审计日志查看接口本身也属于敏感能力，必须受独立权限 `operation_log:view` 控制，并遵守数据范围限制
- 高危操作前端必须展示二次确认信息，后端不得仅依赖前端弹窗；未提供必要原因或确认信息时必须拒绝执行
- 任何后台高危操作设计变更都必须同步检查权限、审计、导出、配置变更、数据范围与日志保留规则是否一致

后续扩展（引入开放平台时）：
若后续建设开放平台（第三方接入、ISV 授权），再引入 AppKey/AppSecret + HMAC-SHA256 签名体系，届时补充：
- 签名字段组成（timestamp + nonce + sorted_params + secret）
- 客户端密钥下发与保管规范
- 时钟漂移容忍（建议 ±5 分钟）
- nonce 存储与过期策略（Redis，TTL 10分钟）

8.3 内容安全
--------------------------------------------------

【违禁词过滤】
- 实时检测聊天内容、评论、职位描述
- 支持精确匹配和模糊匹配
- 处理动作：过滤（替换***）/拦截（拒绝发送）/警告（通知管理员）
- Redis缓存违禁词库，动态更新
- 管理员后台配置

【联系方式过滤】
- 当 contact_filter_enabled=1 时开启
- 过滤范围：私聊消息、公共聊天室消息
- 过滤规则由 contact_filter_pattern 配置
- 默认过滤：手机号格式（1[3-9]\d{9}）、微信/QQ相关词汇
- 触发时返回错误码 7004

【举报机制】
- 用户举报入口（职位、评论、用户均可举报）
- 管理员审核处理
- 违规内容下架
- 用户拉黑公示

8.4 性能优化

- 职位列表缓存（TTL: 5分钟）
- 工种分类缓存（TTL: 1小时）
- 用户信息缓存（TTL: 30分钟）
- 违禁词库缓存（动态更新）
- 图形验证码（TTL: 5分钟）
- 职位浏览防刷（TTL: 5分钟，key: job_view_dedupe:{job_id}:{viewer_key}）

【CDN加速】
- 静态资源CDN分发
- 图片CDN加速
- 就近访问节点

【前端优化】
- 图片懒加载
- 分页加载
- 资源压缩
- 代码分割

【WebSocket优化】
- 心跳机制：客户端每30秒发送ping，服务端响应pong；连续90秒无响应判定为断线
- 断线重连：指数退避策略 1s/2s/4s/8s/16s，第6次起固定30秒间隔重试，最多重试10次
- 主动关闭：客户端发送 leave.conversation（可选）后正常关闭连接，不触发重连
- 异常断开：网络故障/心跳超时导致连接断开，自动按重连策略重试
- 消息队列（防止高并发丢消息）
- 负载均衡

【Laravel 工程化建议】
- 测试：核心接口建议补齐 Feature Test，复杂状态流转（报名审核、企业认证、黑名单处理、职位发布）建议补齐 Unit / Feature 混合测试
