### 问题1：报名表is_latest与状态流转的语义冲突
**所在章节**：3.2 详细表结构设计 - applications表  
**原文要点**：
> - 【is_latest=1的有效状态范围】is_latest=1时status只能为applied/accepted/rejected；cancelled状态的报名is_latest必须为0（取消报名时同步置0）

**问题描述**：
取消报名流程(5.6)明确指出状态可流转为cancelled，但在"报名状态流转统一为：applied -> accepted / rejected / cancelled"的设计中，当is_latest=1时不允许status=cancelled，这与取消操作产生了语义冲突。

**冲突说明**：
- 流程要求：取消报名时 `status = cancelled, is_latest = 0`
- 状态机要求：is_latest=1时status只能是applied/accepted/rejected

这两个规则在"取消最新有效报名"时是一致的，但文档表述容易让人误解为is_latest=1时不能有cancelled状态。

**影响**：文档一致性模糊，可能导致开发理解偏差

**建议修正**：
在3.2节applications表说明中增加澄清：
> 取消报名时必须将is_latest置0（因为cancelled状态不是"有效"状态），但可以保留当前记录供参考。重新报名时is_latest=1的新记录状态为applied。

**关联修改位置**：
- 3.2节applications表说明
- 5.6节取消报名流程

### 问题15：职位发布审核规则中的"认证时间≤30天"判断逻辑不完整
**所在章节**：5.3 职位发布流程  
**原文要点**：
> 企业认证时间超过30天 → status = active，audit_status = 1（直接发布，视为系统自动审核通过）
> 企业认证时间≤30天：
>   ├─ 统计该企业认证通过后发布的职位数量（status != 'draft'）
>   ├─ 数量 < first_jobs_audit_limit（默认10） → status = pending
>   └─ 数量 >= first_jobs_audit_limit → status = active

**问题描述**：
审核规则中"认证时间"的概念不明确：

### 问题23：users表is_deleted与is_blacklist的优先级规则描述不清晰
**所在章节**：3.2 详细表结构设计 - users表  
**原文要点**：
> 【账号状态优先级】（从高到低）：1)已注销(is_deleted=2) 2)账号锁定(login_locked_until>now()) 3)黑名单(is_blacklist=1) 4)注销申请中(is_deleted=1) 5)正常

**问题描述**：
优先级描述中"账号锁定"使用`login_locked_until>now()`判断，但这是一个时间条件，不是固定状态。

**冲突说明**：
- `login_locked_until`是一个时间字段，过了该时间就自动解锁
- 但优先级列表将其与"固定状态"并列，容易混淆

**影响**：状态判断逻辑实现可能出错

**建议修正**：
将账号锁定视为一个计算后的状态：
> 【账号状态优先级计算规则】（从高到低）：
> 1. **已注销** (is_deleted=2) - 最高优先级，账号已永久删除
> 2. **临时锁定** (login_locked_until > 当前时间) - 由login_locked_until字段计算得出
> 3. **黑名单** (is_blacklist=1) - 被平台封禁
> 4. **注销申请中** (is_deleted=1) - 冷静期内
> 5. **正常** - 无以上任何限制
>
> 计算示例：
> ```
> if (is_deleted == 2) return '已注销';
> if (login_locked_until != null && login_locked_until > now()) return '临时锁定';
> if (is_blacklist == 1) return '黑名单';
> if (is_deleted == 1) return '注销申请中';
> return '正常';
> ```

**关联修改位置**：
